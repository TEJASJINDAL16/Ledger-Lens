## ENGINEERING RULES

**Configuration**
- Every tunable number lives in `config/pipeline.yaml`. No magic numbers in code — not thresholds, not window sizes, not injection rates, not file paths.
- Config is loaded once and passed down. No module reads the YAML on its own.

**Data safety**
- No row is ever dropped silently. Anything rejected goes to `data/quarantine/` with a `reject_reason` column.
- Never overwrite a source column. Cleaning adds new columns (`amount_inr` alongside `amount_original`), it does not destroy the original value.
- Any function that changes row count logs the before and after counts.

**Testing**
- Write tests in the same phase as the code they test, not at the end.
- The merchant resolver gets a golden set of 30 hand-written `descriptor -> expected_merchant` pairs covering every corruption transform. This is the regression suite.
- Edge cases in the parsers are non-negotiable: `"(1,240.00)"`, `"₹ 18.99 "`, `"-999"`, `""`, epoch integers.

**Reproducibility**
- Seed every random operation. Two runs of `make all` on a clean checkout must produce identical outputs.
- `make all` runs the entire pipeline from scratch in one command.

## CODE STYLE

Write this the way a competent engineer writes it when nobody is watching. Specifically:

**Naming**
- Modules and functions: `snake_case`. Classes: `PascalCase`. Constants: `UPPER_SNAKE`.
- Functions are `verb_noun`: `parse_amount`, `resolve_merchant`, `flag_near_duplicates`, `load_warehouse`. Not `do_thing`, not `process_data`, not `handler`.
- Use the domain vocabulary from the spec consistently and everywhere: *descriptor*, *canonical merchant*, *quarantine*, *gate*, *tier*, *bronze/silver/gold*, *confidence*, *defect class*. Pick the term once and never drift to a synonym.
- No cute or abstract names. No `Manager`, `Helper`, `Utils`, `Processor`, `Engine` as a class name unless it genuinely is one.

**Comments and docstrings**
- Comment the *why*, never the *what*. `# Toast and Square prepend the merchant name with an aggregator tag` is useful. `# loop over the rows` is noise.
- One concise docstring per module and per non-obvious function: what goes in, what comes out, which defect classes it handles. Do not docstring one-line helpers.
- Do not narrate the code. Do not leave section-divider comment banners.
- No TODO/FIXME left behind unless it is a real deferred decision, and then it must also appear in `docs/DECISIONS.md`.

**General**
- Type hints on public functions. Not on every local variable.
- Prefer boring, explicit code over clever one-liners.
- Do not over-engineer. No abstract base classes, no plugin registries, no dependency injection framework. This is a pipeline, not a platform.
- It is fine — good, actually — if the code has the small asymmetries of real work: a slightly longer function where the logic warrants it, a pragmatic shortcut with a one-line comment explaining the tradeoff.
